# v2.5.0 - Advanced Event Schema Governance

## Summary

v2.5.0 makes TradeOps event contracts explicit, versioned, documented, and easier to validate while keeping the platform lightweight and backward-compatible.

## Added

- Versioned JSON Schemas under `schemas/events/` for core Kafka/Redpanda event topics.
- Common schemas for the event envelope, DLQ messages, and WebSocket stream messages.
- `schemas/events/sample-mapping.json` for validating existing demo samples against event schemas.
- `scripts/validate-event-schemas.sh` and `scripts/validate-event-schemas.js`.
- Event governance documentation:
  - `docs/events/schema-governance.md`
  - `docs/events/event-envelope.md`
  - `docs/events/compatibility-rules.md`
  - `docs/events/event-catalog.md`
  - `docs/events/event-contract-checklist.md`

## Updated

- README with event schema governance and validation instructions.
- Documentation index with event contract links.
- Architecture, event flow, WebSocket, DLQ, tracing, and multitenancy docs with schema/envelope references.
- Makefile and CI shell validation to include the event schema validator.

## Compatibility

- No breaking event payload changes.
- Schemas allow current flat event payloads.
- Envelope metadata such as `eventVersion`, `tenantId`, `correlationId`, and `traceparent` remains additive and optional where existing producers do not emit it yet.
- Consumers are expected to continue tolerant parsing and ignore unknown fields.

## Known Limitations

- No real Kafka/Redpanda Schema Registry service is added in this release.
- Runtime producer/consumer schema enforcement is not enabled.
- Optional Ajv-based validation runs only when Ajv is installed; otherwise the validator performs JSON parsing and mapping checks.
- Some producers already emit `eventType`, `tenantId`, `correlationId`, and tracing metadata; full `eventVersion` coverage is documented as a future hardening step.
