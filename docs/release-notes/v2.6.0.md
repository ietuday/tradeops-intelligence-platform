# v2.6.0 - Rule Configuration Management

## Summary

v2.6.0 makes surveillance rule thresholds and enable/disable state configurable through tenant-aware, RBAC-protected APIs backed by PostgreSQL.

## Added

- `surveillance_rule_configs` database table and service-local migration.
- Central demo migration and idempotent seed for default surveillance rule configs.
- Tenant-aware rule config domain model, repository, service, validation, and in-memory cache refresh.
- Rule management APIs under `/api/v1/surveillance/rules`.
- Enable/disable support for existing surveillance rules.
- Rule config change events:
  - `surveillance.rule_config.updated`
  - `surveillance.rule_config.enabled`
  - `surveillance.rule_config.disabled`
- Event schemas for rule config change events.
- Prometheus metrics for rule config updates, reloads, disabled-rule skips, and cache entries.
- Safe demo script: `scripts/demo-rule-config.sh`.
- Documentation for rule config APIs, fallback behavior, RBAC, events, metrics, and demo flow.

## Compatibility

- Existing environment variables remain fallback defaults.
- Existing alert behavior remains backward-compatible if the rule config table is empty or unavailable.
- No new service, frontend, rules engine, Kubernetes, or Helm changes.

## Known Limitations

- Only built-in surveillance rules are configurable.
- Rule config change events are published, but audit-service normalization for these topics is a future extension.
- Cache refresh is simple and local to the service instance.
