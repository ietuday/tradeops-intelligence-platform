# v2.2.0 - Multi-Tenant Architecture

## Summary

v2.2.0 adds tenant-aware architecture across identity, gateway propagation, tenant-owned database tables, events, audit records, demo payloads, and WebSocket streams. The platform continues to use shared PostgreSQL databases with `tenant_id` columns rather than database-per-tenant.

## Added

- Multitenancy docs under `docs/multitenancy/`.
- Default tenant strategy using `default-tenant`.
- `tenantId` JWT claim and `/me` response support.
- `X-Tenant-ID` HTTP propagation through API Gateway.
- Additive `tenant_id` migrations for tenant-owned tables.
- Tenant-aware event fields for order, portfolio, surveillance, notification, and audit events.
- Tenant-aware WebSocket event filtering.
- Tenant IDs in sample payloads and demo scripts.

## Known Limitations

- Market data remains global by default.
- Strategy and risk tenant filtering is intentionally minimal in this release.
- Tenant IDs are not Prometheus labels to avoid high-cardinality metrics.
- Future hardening can add `NOT NULL` tenant constraints, tenant-specific rate limits, tenant-specific encryption keys, partitions, or schema/database-per-tenant isolation.

## Compatibility

- Existing demo data falls back to `default-tenant`.
- No database-per-tenant migration is introduced.
- No breaking REST API changes are intended.
